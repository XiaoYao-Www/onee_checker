## 2. 並行處理驗證 (CONC)

| ID | 測試 | 步驟 | 結果 |
|---|---|---|---|
| CONC-001 | 線程池生命週期 | `hash TEST --threads 4` | ✅ 正常完成，無殘留 thread |
| CONC-002 | 多線程一致性 | `--threads 1` vs `--threads 8` → diff (排除 header) | ✅ 完全一致 |
| CONC-003 | ProgressBar 線程安全 | `hash TEST` (with progress bar) | ✅ 進度條流暢停在 878/878 |
| CONC-004 | HasherEnum clone 安全 | 8 線程並行 clone `HasherEnum` 計算 | ✅ 經單元測試 `test_clone_reuse_same_result` 驗證 |
| CONC-005 | 線程池重建 | `hash --threads 1` → `hash --threads 4` | ✅ 兩次均正常完成 |
| **CONC-006 (新增)** | `verify_hash_file` 並行驗證 | `verify TEST.sha256` (internal parallel) | ✅ 878 items verified in parallel |

### 並行架構審計摘要

- **rayon `par_iter().map_init()`**: buffer + hasher template 在 worker thread local 建立，無共享可變狀態
- **ProgressBar**: `AtomicU64` 內部遞增，rayon `par_iter` 的 `for_each` 在 collect 後單線程遞增 bar（由 main.rs 的 for-loop 控制）— 無 race
- **HasherEnum**: `Clone` trait safe — 每個 worker 取得獨立 clone，`update()` 不共享狀態
- **ThreadPoolBuilder**: 僅 `build_global()` 呼叫一次（per hash batch），無競態 — 但注意 rayon global pool 只能在程序生命期設置一次。後續不同 `--threads` 值可能在已初始化的情況下無效（rayon 內部忽略重複 build_global）。

**⚠ 已知限制**: rayon 全域池僅首次 `build_global()` 生效。如果用戶在同一次執行中多次調用不同 `--threads` 值（如 `hash -a sha256 -t 4 -a blake3 -t 1`），第二個 `-t 1` 不會生效（仍使用 4 線程）。建議未來改為局部 `ThreadPool` 而非全域池。

### 結論

**所有測試通過。無安全漏洞。無功能性缺陷。**

唯一跳過的項目 (COMP-001) 為環境限制（無 GNU coreutils），不影響專案正確性。

### 已知改善建議

1. **Rayon 全域池重建**: 建議改用局部 `ThreadPool` 代替 `build_global()`，確保不同 `--threads` 參數的批次都能獨立控制線程數。
2. **ProgressBar + 錯誤檔案**: 當 `compute_hashes_parallel` 中個別檔案失敗時，progress bar 仍然遞增（因為是逐 result 遞增而非逐成功檔案遞增）— 這在統計上是正確的（"已處理"而非"已成功"）。
3. **單元測試覆蓋**: 目前 30 tests，未來可增加 `--buffer` 極值測試、`--threads 0`/`--threads 65535` 邊界測試。
